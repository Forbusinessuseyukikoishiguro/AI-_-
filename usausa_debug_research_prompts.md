## 土台にした論文 4本

### 📄 論文① Chain-of-Thought Prompting（CoT）
**Wei et al., NeurIPS 2022**  
https://arxiv.org/abs/2201.11903

「中間の推論ステップを順番に書かせる」だけで、LLMの複雑な問題解決能力が大幅に向上することを示した論文。デバッグ分析で「なぜ？」を段階的に言語化させる根拠になっています。

> *"generating a chain of thought — a series of intermediate reasoning steps — significantly improves the ability of large language models to perform complex reasoning"*

---

### 📄 論文② Self-Debugging（自己デバッグ）
**Chen et al., arXiv 2023（ICLR 2024採択）**  
https://arxiv.org/abs/2304.05128

LLMが「生成したコードを自分で説明しながらバグを発見・修正する」フレームワーク。エラーメッセージなしでも、**コードを一行ずつ自然言語で説明させるだけ**でバグを発見できることを示した。

> *"without any feedback on the code correctness or error messages, the model is able to identify its mistakes by explaining the generated code in natural language"*

---

### 📄 論文③ LDB（Debug like a Human）
**Zhong, Wang & Shang, ACL 2024 Findings**  
https://arxiv.org/abs/2402.16906  
（ACL Anthology: https://aclanthology.org/2024.findings-acl.49/）

人間のデバッガが「ブレークポイントを置いて変数の中間値を確認する」ように、コードをブロックに分解して**実行トレース＋中間変数を追わせる**手法。HumanEval/MBPP/TransCoderで最大9.8%の精度向上。

> *"LDB segments the programs into basic blocks and tracks the values of intermediate variables after each block throughout the runtime execution"*

---

### 📄 論文④ The Prompt Report（プロンプト手法の体系的サーベイ）
**Schulhoff et al., arXiv 2024**  
https://arxiv.org/abs/2406.06608

58種類のプロンプト手法を分類・分析した現時点で最も包括的なサーベイ論文。「ロールの指定」「出力形式の指定」「Few-shot例示」などの有効性が実証的に整理されている。

## プロンプトの設計原則（論文から導いた3つ）

| 原則 | 由来 |
|---|---|
| **段階的に推論させる**（いきなり答えを求めない） | Wei et al. 2022（CoT） |
| **コードを説明させてからバグを探させる** | Chen et al. 2023（Self-Debugging） |
| **ブロック単位で変数の値を追わせる** | Zhong et al. 2024（LDB） |

## PHASE 1 ── なぜ？（Why）── 原因を探る

> **論文の根拠：** CoT（Wei et al. 2022）は「いきなり答えを求めると推論が浅くなる」ことを示した。「なぜ？」を段階的に言語化させることで、見落としが減る。

---

### 🔍 W-01 ：現状を言語化する（具体レベル）

```
以下の状況を整理して、「現状何が起きているか」を一文で表現してください。

【やりたかったこと（理想の動作）】


【実際に起きたこと（現状の動作）】


【理想と現状のギャップ】


整理したあと、以下の手順で原因を探ってください：
ステップ1：この症状が起きる「直接的な原因」の候補を3つ挙げる
ステップ2：それぞれの候補を「確認するにはどうすればいいか」を書く
ステップ3：最も可能性が高い候補を1つ選び、その理由を述べる

（根拠：Wei et al. 2022 Chain-of-Thought：推論を段階的に明示させることで精度が向上する）
```

---

### 🔍 W-02 ：コードを「説明させて」バグを探す

```
以下のコードを一行ずつ自然言語で説明してください。
説明しながら、「意図と実際の動作がズレている行」を探してください。

【コード】


説明の形式：
行N: （その行が何をするか）→ 【注意】意図とズレている場合はここに書く

全行説明が終わったら：
■ 意図とズレていた行：
■ そのズレが起きる理由：
■ 修正の方向性：

（根拠：Chen et al. 2023 Self-Debugging：コードを自然言語で説明させると、
エラーメッセージなしでも LLM がバグを自己発見できる）
```

---

### 🔍 W-03 ：ブロック単位で変数の値を追う

```
以下のコードを「ブロック（処理のまとまり）」に分解して、
各ブロックの終わりでの変数の値を追ってください。

【コード】


【追いたい変数名】


分析形式：
ブロック1（行N〜M）: 何をしているか
　→ 終了時点での【変数名】の値: 
ブロック2（行N〜M）: 何をしているか
　→ 終了時点での【変数名】の値: 
…

「ここで値がおかしくなっている」ブロックを特定したら、その原因を説明してください。

（根拠：Zhong et al. 2024 LDB：プログラムをブロックに分解し中間変数を追うことで、
複雑な制御フローのバグを効率的に特定できる）
```

> 💡 **うさうさメモ（経験知・論文外）**  
> 今回の読書手帳バグで言うと、「weekDone をリセットした直後の値」と「renderWeek が呼ばれたときの値」の2ブロックを追えば、どこで上書きされたかが見えます。

---

### 🔍 W-04 ：抽象パターンで分類する

```
以下のバグを「コードの具体的な間違い」ではなく、
「設計・パターンレベルの問題」として分類してください。

【バグの症状】


【関係するコード全体】


以下のパターンから当てはまるものを選び、なぜそのパターンかを説明してください：

A. 状態の競合　── 複数箇所から同じ変数を書き換えている
B. 破壊的変更　── 元のデータが意図せず変わっている（例：Date.setDate()）
C. 参照の共有　── 同じオブジェクトを指していて片方を変えると両方変わる
D. タイミングのズレ── 非同期・タイマー・イベントの実行順序が意図と違う
E. 環境依存　　── ブラウザ・iframe・オリジンの違いで挙動が変わる
F. 責務の混在　── 1つの関数が複数のことをしていて副作用が出ている

パターンを選んだ理由と、そのパターンに共通する「直し方の方向性」を述べてください。
```

---

### 🔍 W-05 ：「なぜ直らないのか」を分析する

```
修正したのに、まだ同じ問題が起きています。
「なぜ修正が効いていないか」を段階的に分析してください。

【直そうとした原因（仮説）】


【加えた修正】


【まだ起きている症状】


ステップ1：修正は「本当の原因」に対処しているか、それとも「症状」に対処しているか？
ステップ2：同じ問題を引き起こしている箇所が他にもないか？
ステップ3：修正が「実際に動いているコード」に反映されているか？
　　　　　（キャッシュ・古いファイル・環境の問題）
ステップ4：修正によって新しい問題が生まれていないか？

「まだ直らない理由」として最も可能性が高いものを1つ選んでください。
```

## PHASE 2 ── 現状と理想のギャップを図解する

> **論文の根拠：** Schulhoff et al. 2024（The Prompt Report）は、「出力形式を明示的に指定する」ことでLLMの回答品質が向上することを整理している（§ Output Format指定の有効性）。

### 📊 G-01 ：現状・理想・ギャップを表で整理する

```
以下の状況について、「現状」「理想」「ギャップの原因」を表形式で整理してください。

【状況説明】


出力形式（表）：

| 観点 | 現状（実際に起きていること） | 理想（本来こうあるべき） | ギャップの原因（なぜ差があるか） |
|------|--------------------------|----------------------|-------------------------------|
| データの状態 | | | |
| 処理の順序 | | | |
| 関数の責務 | | | |
| 環境・設定 | | | |

表を埋めたあと、「最も根本的なギャップ」を1行で述べてください。
```

---

### 📊 G-02 ：処理フローを時系列で図示する

```
以下のコードの処理フローを、時系列の図（テキストで）として表してください。

【コード】


出力形式：

時刻T1: 【関数名】が呼ばれる → 【変数名】= 【値】
時刻T2: 【関数名】が呼ばれる → 【変数名】= 【値】
…

「理想の処理フロー」と「実際の処理フロー」を並べて書いてください。
ズレている箇所に ⚠️ を付けてください。

理想：
T1: ...
T2: ...

実際：
T1: ...
T2: ... ⚠️（ここでズレ発生）
```

---

### 📊 G-03 ：競合している処理を洗い出す

```
「同じ状態を書き換えている処理」をすべて洗い出して、競合マップを作ってください。

【調べたい状態・変数】


【コード全体（または関係する部分）】


出力形式：

■ 【変数名】を書き換えている箇所：

| # | 関数名 | 書き換え内容 | 実行タイミング |
|---|--------|------------|-------------|
| 1 | | | |
| 2 | | | |

■ 競合が起きるシナリオ：
（どの順番で実行されたときに、意図と反する上書きが起きるか）

■ 「どれを正とするか」の提案：
```

## PHASE 3 ── どう直す？（How to Fix）

> **論文の根拠：** Chen et al. 2023（Self-Debugging）は、「修正の前にコードを説明させる」ことで修正精度が上がることを示した。「直す前に理解する」ステップが重要。

---

### 🛠 F-01 ：修正方針を「応急処置 vs 根本解決」で分ける

```
以下のバグについて、修正方針を2種類提案してください。

【バグの原因（特定済み）】


【関係するコード】


■ 応急処置（今すぐ動かすための最小限の修正）
- 変更する箇所：
- 変更内容：
- リスク・副作用：
- 所要時間の目安：

■ 根本解決（設計レベルで正しく直す）
- 変更する箇所：
- 変更内容：
- リスク・副作用：
- 所要時間の目安：

どちらを選ぶべきか、判断基準も述べてください。
```

---

### 🛠 F-02 ：修正の影響範囲を確認してから直す

```
以下の修正を加える前に、影響範囲を確認してください。

【変更しようとしているコード】


【変更内容】


確認してほしいこと：
① この変更によって動作が変わる可能性がある他の関数・処理
② 変更後に壊れる可能性があるもの
③ 変更前に確認すべき「前提条件」

「影響が少ない順」に修正の進め方を提案してください。
修正の各ステップに「確認ポイント」もつけてください。
```

---

### 🛠 F-03 ：同じパターンのバグが他にないか探す

```
以下のバグパターンが、コードの他の箇所にも潜んでいないか探してください。

【見つかったバグのパターン】
（例：Dateオブジェクトを setXxx() メソッドで破壊的に変更している）

【コード全体】


① 同じパターンが当てはまる箇所をすべてリストアップ
② 優先度が高い順に並べる（ユーザーへの影響が大きいもの順）
③ まとめて修正できる共通の直し方があれば提案する

見つからなかった場合は「見つからなかった」と正直に書いてください。
```

## PHASE 4 ── 試す（Verify）── 直ったかを確認する

> **論文の根拠：** Chen et al. 2023（Self-Debugging）は「実行フィードバックを使った反復的な検証」の重要性を示した。「直したつもり」ではなく「直ったことを確認した」が正しい終わり方。

---

### ✅ V-01 ：確認すべきテストケースを列挙する

```
以下の修正が正しく動いているか確認するためのテストケースを作ってください。

【修正した内容】


【修正前に起きていたバグ】


テストケースの形式：

| # | 操作（何をするか） | 期待する結果 | 確認方法 |
|---|----------------|------------|--------|
| 1 | | | |
| 2 | | | |

特に以下を必ず含めてください：
- バグが再発しやすいケース
- 境界値（最小・最大・0・null）
- 修正した関数を複数回連続して呼んだ場合
- 修正前は動いていた処理がまだ動くか（デグレ確認）
```

---

### ✅ V-02 ：ログで確認すべき場所を指定してもらう

```
以下のバグを調査するために、どこにログ（console.log等）を仕込めばいいか教えてください。

【調査したい症状】


【関係するコード】


以下を指定してください：
① どの関数の、どの行の「前後」に入れるか
② 何を出力すれば原因を絞り込めるか
③ 出力結果の「正常な場合の値」と「異常な場合の値」の違い

実際に貼れるログのコードも書いてください。

（根拠：Zhong et al. 2024 LDB：中間変数の値を追うことで複雑なバグを効率的に特定できる）
```

---

### ✅ V-03 ：「直った」の判断基準を定義する

```
修正後、「バグが直った」と判断するための基準を定義してください。

【修正したバグ】


【修正内容】


出力形式：

■ 直ったと言える条件（すべて満たす必要がある）
1.
2.
3.

■ 直っていないと判断するサイン

■ 「動いているように見えるが実は直っていない」パターン

■ 確認の順番（何を先に確認するか）
```

---

### ✅ V-04 ：修正後のコードをレビューする

```
バグ修正後のコードをレビューしてください。

【修正前のコード】


【修正後のコード】


【直そうとしたバグ】


確認してほしいこと：
① 修正は「本当の原因」に対処しているか
② 修正によってデグレ（新しいバグ）が発生していないか
③ エッジケース（null・最大値・連続操作）でも動くか
④ 同じパターンのバグが他の箇所に残っていないか

問題が残っている場合は「まだ直っていない理由」を具体的に指摘してください。
問題がない場合も「問題なし」とはっきり書いてください。
```

## 組み合わせ例：今回の読書手帳バグ（実体験）

実際にどのプロンプトを使えばよかったかを再現します。

```
【バグ】リセットボタンを押しても、桜（weekDone）が消えない

① W-01（現状を言語化）
  →「週リセット処理を呼んでいるはずなのに画面が更新されない」
  →「状態管理のバグ」と分類

② W-04（抽象パターンで分類）
  → Aパターン：「複数箇所から weekDone を書き換えている」と特定

③ G-03（競合マップ）
  → checkWeek()・resetWeekDone()・morningBtn の3箇所が書き換えていると判明

④ W-03（ブロック単位で変数を追う）
  → weekKey() の中で d.setDate() が d を破壊的に変更していることを発見

⑤ F-01（修正方針）
  → 応急：weekKey() を new Date() で別オブジェクト作成に変更
  → 根本：状態管理を一箇所に集約する設計に見直す

⑥ V-01（テストケース）
  → 「桜を咲かせた後にリセット」
     「リセット後に朝の一服を押す（今日分だけ咲くか）」
     「ページ再読み込み後もリセット状態が保持されるか」

⑦ V-03（直った基準）
  → 全7日が🪷に戻り、30秒待っても元に戻らないこと
```
