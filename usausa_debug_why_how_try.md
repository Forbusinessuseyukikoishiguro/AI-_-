## この記事の使い方

バグ解決は **3ステップの繰り返し** です。

```
①「なぜ？」  ── 原因を特定する（具体 ⇄ 抽象を行き来する）
      ↓
②「どうすれば？」 ── 対策を考える
      ↓
③「試す」 ── 動作確認する
      ↓
まだ直らない → ①に戻る
```

このプロンプト集は、そのフローに沿って使います。  
`【】`の中を自分の状況に書き換えてください。

---

## STEP 1 ── なぜ？（原因を探る）

> ポイント：「具体」と「抽象」の両方から攻める。  
> 具体だけ見ると「この行がおかしい」で終わる。抽象で見ると「この設計パターンがそもそも危険」に気づける。

---

### 🔍 W-01 ：まず「何が起きているか」を言語化する

```
次の状況を整理して、「何が起きているか」を一言で表現してください。

【やりたかったこと】


【実際に起きたこと】


【気になっている点】


整理したあと、「これはどういうカテゴリのバグか」を以下から選んで理由も教えてください：
- 状態管理のバグ（値が意図せず変わる・変わらない）
- タイミングのバグ（処理の順序がズレている）
- 環境のバグ（特定の環境でしか起きない）
- 論理のバグ（条件分岐や計算が間違っている）
- 表示のバグ（値は正しいが画面に反映されない）
```

---

### 🔍 W-02 ：具体レベルで原因を絞る

```
以下のコードで「【症状】」が起きています。
コードを一行ずつ追って、どこで意図と外れているかを特定してください。

【症状】


【コード】


分析は以下の形式でお願いします：

■ 処理の流れ（実際に何が起きているか、順番に）
1.
2.
3.

■ 「ここが怪しい」という行とその理由

■ 断言できる原因 or まだ絞り切れていない場合は「次に確認すべきこと」
```

---

### 🔍 W-03 ：抽象レベルで原因を見る

```
以下のバグを、「コードの具体的な間違い」ではなく
「設計・パターンレベルの問題」として分析してください。

【バグの症状】


【関係するコード全体】


抽象レベルで見たとき、このバグは次のどのパターンに当てはまりますか？

A. 状態が複数箇所から書き換えられている（競合・上書き）
B. 破壊的変更（元のデータが意図せず変わる）
C. 参照の共有（同じオブジェクトを指していて片方を変えると両方変わる）
D. 実行タイミングのズレ（非同期・イベント・タイマー）
E. 環境依存（ブラウザ・iframe・オリジンの違い）
F. 関数の責務の混在（1つの関数が複数のことをしている）

当てはまるパターンを選び、「なぜそのパターンが起きたか」を説明してください。
```

> 💡 **うさうさメモ**  
> 今回の読書手帳バグは A（`checkWeek()` と `resetWeekDone()` が `weekDone` を両方書き換えていた）＋ B（`Date.setDate()` が元オブジェクトを破壊的に変更していた）の複合でした。抽象で見ると「Aパターン」と気づければ、コードを全部読まなくても「状態を書き換えている箇所をすべて探す」という正しい手が打てます。

### 🔍 W-04 ：「なぜ直らないのか」を分析する

```
修正したのに、まだ同じ問題が起きています。
「なぜ修正が効いていないか」を分析してください。

【直そうとした原因】


【加えた修正】


【まだ起きている症状】


次の観点で確認してください：

① 修正は「本当の原因」に対処しているか、それとも「症状」に対処しているか
② 同じ問題を引き起こしている箇所が他にもないか
③ 修正によって新しい問題が生まれていないか
④ 修正が実際に動いているコードに反映されているか（キャッシュ・古いファイルの問題）

「まだ直らない理由」を最も可能性が高い順に3つ挙げてください。
```

## STEP 2 ── どうすれば？（対策を考える）

> ポイント：「動けばいい」と「正しく直す」は違う。  
> 応急処置か根本解決かを意識して選ぶ。

---

### 🛠 H-01 ：修正の方針を決める

```
以下のバグについて、修正方針を考えてください。

【バグの原因（特定済み）】


【関係するコード】


修正の選択肢を以下の2軸で整理してください：

■ 応急処置（今すぐ動かすための最小限の修正）
- やること：
- リスク：
- 所要時間：

■ 根本解決（設計レベルで正しく直す）
- やること：
- リスク：
- 所要時間：

どちらを選ぶべきか、状況に応じた判断基準も教えてください。
```

---

### 🛠 H-02 ：影響範囲を確認してから直す

```
以下の修正を加えようとしています。
変更の影響範囲を確認してください。

【変更しようとしているコード】


【変更内容】


確認してほしいこと：
① この変更が影響する他の関数・処理はどこか
② 変更によって動かなくなる可能性があるものは何か
③ 変更が安全かどうかを確認するために、事前にチェックすべき項目

「影響が少ない順」に修正の進め方を提案してください。
```

---

### 🛠 H-03 ：同じパターンのバグが他にもないか探す

```
以下のバグパターンが、コードの他の箇所にも潜んでいないか探してください。

【見つかったバグのパターン】
（例：Dateオブジェクトをsetメソッドで破壊的に変更している）

【コード全体】


同じパターンを検索して：
① 同じ問題が起きている箇所をすべてリストアップ
② 優先度が高い順に並べる（ユーザーへの影響が大きいもの順）
③ まとめて修正できる共通の直し方があれば提案する
```

> 💡 **うさうさメモ**  
> `weekKey()` の `d.setDate()` バグを直したとき、「他の日付処理でも同じことをしていないか」を確認するのがプロのやり方です。1つ見つかったら「同じパターンが他にもある」と疑う習慣を持ちましょう。

---

### 🛠 H-04 ：競合している処理を整理する

```
複数の処理が同じ状態を書き換えていて、競合が起きています。
整理して解決策を出してください。

【書き換えられている状態】
（例：state.weekDone という配列）

【書き換えている処理をすべて列挙】
1. （処理名・コード・実行タイミング）
2.
3.

それぞれが「いつ実行されるか」を時系列で並べて、
どの順番で実行されたときに競合が起きるかを図示してください。

解決策は「どこを正とするか」を決めるところから始めてください。
```

## STEP 3 ── 試す（動作確認する）

> ポイント：「動いた」ではなく「期待通りに動いたことを確認した」が正しい確認。

---

### ✅ T-01 ：確認すべきテストケースを出してもらう

```
以下の修正が正しく動いているか確認したいです。
確認すべきテストケースを出してください。

【修正した内容】


【修正前に起きていたバグ】


テストケースは以下の形式で5〜10個出してください：

| # | 操作 | 期待する結果 | 確認方法 |
|---|------|------------|--------|
| 1 | | | |

特に「バグが再発しやすいケース」「境界値」「連続操作」を含めてください。
```

---

### ✅ T-02 ：console.logで確認すべき場所を教えてもらう

```
以下のバグを調査するために、どこにconsole.logを仕込めばいいか教えてください。

【調査したい症状】


【関係するコード】


以下を指定してください：
① どの関数の、どの行の「前後」に入れるか
② 何を出力すれば原因を絞り込めるか
③ 出力結果のどこを見ればいいか（正常な場合と異常な場合の違い）

実際に貼れるconsole.logのコードも書いてください。
```

---

### ✅ T-03 ：「直ったかどうか」を判断する基準を決める

```
修正後、「バグが直った」と判断するための基準を決めてください。

【修正したバグ】


【修正内容】


以下を定義してください：

■ 直ったと言える条件（全部満たす必要があるもの）
1.
2.
3.

■ 直っていないと判断するサイン

■ 「動いているように見えるが実は直っていない」パターン

確認の順番（何を先に確認するか）も教えてください。
```

---

### ✅ T-04 ：デグレ（新しいバグ）が出ていないか確認する

```
修正を加えました。修正によって新しい問題が発生していないか確認してください。

【修正前のコード】


【修正後のコード】


【修正の意図】


デグレが起きていないか、以下の観点で確認してください：
① 修正した関数を呼んでいる他の箇所で動作が変わっていないか
② 修正によって削除・変更されたコードに依存していた処理がないか
③ エッジケース（空の値・最大値・連続操作）で問題が出ていないか

問題があれば「何が壊れたか」と「どう直すか」をセットで教えてください。
```

## 組み合わせ例：今回の読書手帳バグの場合

実際に今回のデバッグで使うべきだったフローを再現します。

```
【バグ】リセットボタンを押しても、桜（weekDone）が消えない

① W-01 で整理
  → 「状態管理のバグ：値が変わらない」と分類

② W-03 で抽象分析
  → 「Aパターン：状態が複数箇所から書き換えられている」と気づく

③ H-04 で競合を整理
  → checkWeek()・resetWeekDone()・morningBtn・finishBtn の4箇所が
    weekDone を書き換えていると判明

④ H-02 で影響範囲を確認
  → weekKey() の修正が checkWeek() の判定に影響することを確認

⑤ T-01 でテストケースを作成
  → 「桜を咲かせてリセット→消えるか」
     「リセット後に朝の一服を押す→今日分だけ咲くか」
     「ページ再読み込み後もリセット状態が保持されるか」

⑥ T-03 で「直った」の基準を定義
  → 全7日分が🪷に戻り、30秒待っても戻らないこと
```

## まとめ：デバッグの思考フロー早見表

| フェーズ | プロンプト | やること |
|---|---|---|
| なぜ？（具体） | W-01, W-02 | 症状を言語化、コードを追う |
| なぜ？（抽象） | W-03, W-04 | パターンで分類、修正が効かない理由を探す |
| どうすれば？ | H-01, H-02 | 応急処置 vs 根本解決を選ぶ |
| どうすれば？（広く） | H-03, H-04 | 他にも同じバグがないか、競合を整理する |
| 試す | T-01, T-02 | テストケースを作り、ログで確認する |
| 試す（判断） | T-03, T-04 | 直った基準を定義し、デグレを確認する |
