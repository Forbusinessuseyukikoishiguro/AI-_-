# 今後のためのプロンプト集 ── 単一HTMLツール開発の進め方

🐰 うさうさ研修工房 ｜ AIに依頼するときの再利用テンプレ

このプロジェクト（オフライン単体HTMLツール）を、品質を保ちながら速く回すためのプロンプト集。`{ }` は埋めて使う。

---

## 0. 前提を一度だけ固定する（最初のメッセージに添える）

毎回の指示を短くできるよう、共通ルールを先に宣言しておくと事故が減る。

```
このプロジェクトの共通ルール：
- 単一HTMLファイル / オフライン / vanilla JSのみ（フレームワーク・外部CDN禁止）
- 既存の見た目と構造はできるだけ壊さない
- 「商用テスト」とは：実ファイルをjsdomで読み込み、ユーザー操作を模擬して検証し、
  TZ=Asia/Tokyo で実行、全アサーションPASSを確認すること
- 破壊的操作（全消去など）はconfirmを挟む
- 嘘をつかない（曖昧な箇所は推測で埋めず、確認する）
以後の依頼はこのルール前提でお願いします。
```

---

## 1. 機能追加 / 仕様変更

```
{機能名}を追加して。
- やりたいこと：{ユーザーから見た振る舞い}
- 受け入れ条件：{こうなったらOK、という具体例を1〜2個}
- 触ってほしくない箇所：{あれば}
実装方針を一言添えてから直して。曖昧なら勝手に決めず、解釈を明示して。
```

仕様変更で「現状と違う挙動」を求めるときは、**現状→あるべき姿**を並べると解釈ブレが消える。

```
今は{現状の挙動}になっている。これを{あるべき挙動}に変えて。
理由は{背景}。副作用が出そうな箇所があれば先に指摘して。
```

---

## 2. テスト & バージョンアップ（今回の定番フロー）

```
いつも通り商用テストして、問題なければバージョンアップして。
- 新機能/変更点に対応するテストケースを追加すること
- 既存機能の非破壊（回帰）も確認すること
- 結果サマリ（PASS/FAIL件数）を見せて
- バージョンはフッター表記とファイル名の両方を更新
```

時刻・日付がからむ変更のときは必ず添える：

```
時刻まわりは TZ=Asia/Tokyo で実行し、境界値（日曜23:59:59ちょうど/直後、月曜、週またぎ）を
独立した参照実装と総当たりで突き合わせて検証して。
```

---

## 3. バグ修正

```
{症状}という不具合がある。
- 再現手順：{1, 2, 3}
- 期待：{こうなってほしい}
- 実際：{こうなる}
原因を特定してから直して。修正後は再現手順をテストケース化して、商用テストに追加して。
```

ログを使う：

```
🩺ログに出ている内容を貼ります：{ログ}
このエラーの原因を切り分けて、根本原因を直して。握りつぶしの暫定対応はしないで。
```

---

## 4. 品質強化 / リファクタ

```
{対象}の堅牢性を強化して（確認強化）。
- 計算/判定が複数箇所に散っていれば単一の真実（純粋関数）に集約
- 例外・失敗時のフォールバックとログ記録を入れる
- 強化点に対応するテストを追加
見た目と既存挙動は変えないこと。
```

設計レビューだけ欲しいとき：

```
このファイルの設計上の弱点を、単一HTML/オフラインの制約を踏まえて指摘して。
直さず、優先度つきの指摘リストだけ先に出して。
```

## 5. ドキュメント生成

```
今回の{ツール名}について、エンジニア向けにマークダウンで：
- 要件定義 / 詳細設計書 / テスト仕様 / 使い方記事
を作って。実装に即して、嘘や誇張なしで。要件→テストがIDで追える形にして。
```

記事（Qiita/Zenn/note）向け：

```
{ツール名}の{使い方/実装/テスト手法}を、エンジニア向けの簡単なマークダウン記事に。
実コードの抜粋を入れて、コピペで動く粒度に。公開先は{Qiita/Zenn/note}なので体裁を寄せて。
```

---

## 6. うまく回すコツ（メタ）

- **受け入れ条件を1〜2個の具体例で渡す**と、解釈ブレと手戻りが激減する。
- **「現状→あるべき姿」**の差分で頼むと、仕様変更が一発で通りやすい。
- **「商用テスト」の定義を最初に固定**しておくと、毎回の指示が「いつも通り」で済む。
- **時刻/日付/週またぎ**は境界値を明示。TZを必ず指定。
- **曖昧さは推測で埋めさせない**。「解釈を明示してから実装」と一言入れる。
- バージョンアップは**ファイル名＋画面表記の両方**を更新対象に含める。

---

## 7. そのまま使える「定番1行」

| 場面 | 1行プロンプト |
|------|---------------|
| 変更後の確認 | いつも通り商用テストして問題なければバージョンアップして |
| 時刻系の確認 | 境界値を参照実装と総当たりで、TZ=Asia/Tokyoで検証して |
| 失敗時の堅牢化 | 失敗時はメモリ退避＋ログにして、入力が消えないようにして |
| 設計集約 | 同じ計算が複数箇所にあるなら単一の純粋関数に寄せて |
| ドキュメント | 実装に即して要件→設計→テストをマークダウンで、IDで追える形に |
